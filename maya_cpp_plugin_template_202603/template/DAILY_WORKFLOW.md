# 日常工作说明

本说明面向实际使用此模板进行 Maya 2026.3 C++ 插件开发的成员，重点说明从拉代码、构建、修改、测试到部署的日常流程。

---

## 1. 第一次接手项目时该做什么

第一次接手时，不要急着增加功能，先确认以下三件事情：

1. 本地能成功生成工程
2. 本地能成功构建 `hello_world_mll`
3. Maya 中能执行 `helloCmd`

只有这三步跑通后，后续开发才有意义。

---

## 2. 每日开始工作前的建议动作

建议每天开始工作时按以下顺序确认：

1. 拉取最新代码
2. 确认当前要处理的插件目录
3. 确认是否涉及公共 cmake 或 shared 代码
4. 确认本次改动会影响哪些插件
5. 再开始修改

如果会改动公共构建规则，最好先和负责人确认。

---

## 3. 开发环境初始化

### 3.1 设置 MAYA_LOCATION

```bat
set MAYA_LOCATION=C:\Program Files\Autodesk\Maya2026
```

### 3.2 生成工程

在 `template/` 目录执行：

```bat
cmake -S . -B build\maya2026_vs2022 -G "Visual Studio 17 2022" -A x64
```

### 3.3 构建 Debug

```bat
cmake --build build\maya2026_vs2022 --config Debug
```

---

## 4. 日常修改代码的推荐顺序

建议按下面顺序工作：

1. 先明确本次需求只影响哪个插件
2. 在插件目录下新建或修改 `.h/.cpp`
3. 更新对应插件的 `CMakeLists.txt`
4. 编译 Debug
5. 进入 Maya 测试
6. 确认输出日志和结果
7. 整理后再提交

不要在没有编译和测试的情况下直接提交。

---

## 5. 如何新增文件

### 5.1 新增命令类

放到：

```text
plugins/<plugin_name>/include/
plugins/<plugin_name>/src/
```

### 5.2 新增节点类

同样放到当前插件自己的 `include/` 和 `src/` 中。

### 5.3 更新构建配置

新文件加入后，要更新当前插件的 `CMakeLists.txt`，否则 Visual Studio 里可能看得到文件，但实际不会参与编译。

---

## 6. 调试时的建议

### 6.1 优先使用 Debug

本地验证阶段优先用 Debug，便于确认：

- 注册是否成功
- 命令是否被调用
- 错误是否能及时输出

### 6.2 保持 plugin_main.cpp 简洁

不要把业务逻辑堆在入口文件中。入口文件只负责：

- 初始化插件
- 注册命令或节点
- 反初始化与注销

这样后续维护会轻松很多。

---

## 7. 在 Maya 中测试的基本流程

### 7.1 加载插件

```python
import maya.cmds as cmds
cmds.loadPlugin("hello_world_mll.mll")
```

### 7.2 执行命令

```python
cmds.helloCmd()
```

### 7.3 如果使用模板附带脚本

```python
import load_hello_plugin
```

然后执行：

```python
import maya.cmds as cmds
cmds.helloCmd()
```

---

## 8. 提交前检查

建议提交前至少检查以下内容：

1. 是否成功编译
2. 是否在 Maya 中完成基本测试
3. 是否误改了公共 cmake
4. 是否把 build 产物加入了版本控制
5. 文档是否需要同步更新

如果改了项目结构或构建方式，必须同步更新说明文档。

---

## 9. 日常发布动作

当你需要把结果发给测试人员或其他用户时，建议按以下顺序：

1. 构建 Release
2. 按 `DEPLOYMENT_WINDOWS11.md` 执行部署
3. 在目标环境验证
4. 再进行交付

不要把 Debug 版直接交给普通用户长期使用。

---

## 10. 遇到 Maya 版本升级时的日常处理方式

如果项目要增加对其他 Maya 版本的支持，开发者不要直接改坏当前版本流程，而应该：

1. 新增新版本构建配置
2. 保留原有版本可用状态
3. 分开验证编译与加载
4. 把问题记录到文档中

这样可以避免一次升级影响整个长期项目。

---

## 11. 当前模板最适合的使用方式

就现在这套模板而言，最适合的使用方式是：

- 先用它快速做出第一个可加载的 mll
- 再逐步新增命令类、节点类和共享代码
- 最后再扩展成真正的多插件长期仓库

这是当前风险最低、最容易落地的路线。
