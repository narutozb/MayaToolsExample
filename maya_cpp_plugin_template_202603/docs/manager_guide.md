# 面向项目管理者的指南

## 1. 适用范围

本指南用于管理一个长期维护的 Maya C++ 插件仓库，重点覆盖以下内容：

- 项目的创建方式
- 文件夹与过滤器管理规则
- 构建策略与构建文件管理
- 依赖导入方法
- 项目属性管理
- 多插件、多项目并行维护方式
- Maya 版本升级时的处理流程

本模板默认面向 **Maya 2026.3 + Visual Studio 2022 + CMake**。

---

## 2. 推荐的仓库管理方式

建议采用 **一个仓库管理多个插件** 的方式，而不是每个 mll 单独一个仓库。

原因如下：

- 可以统一管理 Maya SDK 路径和构建规则
- 多个插件共享公共 cmake 逻辑更稳定
- 更适合长期维护与版本升级
- 更方便管理公共头文件、公共工具代码和统一输出目录

推荐结构如下：

```text
repo_root/
├─ cmake/
├─ plugins/
│  ├─ plugin_a/
│  ├─ plugin_b/
│  └─ plugin_c/
├─ shared/
│  ├─ include/
│  └─ src/
├─ scripts/
├─ docs/
└─ build/
```

其中：

- `cmake/`：统一管理构建规则
- `plugins/`：每个插件一个目录
- `shared/`：多个插件共用的代码
- `scripts/`：测试加载、部署、辅助脚本
- `docs/`：管理文档、开发文档、升级记录
- `build/`：本地构建产物，不入库

---

## 3. 文件夹与过滤器管理规则

### 3.1 文件夹必须与物理路径一致

Visual Studio 的过滤器只作为展示用途，但必须尽量保持与磁盘路径一致。

建议规则：

- `include/` 只放头文件
- `src/` 只放 cpp
- `tests/` 只放测试代码
- `resources/` 只放资源文件
- 插件入口文件命名统一为 `plugin_main.cpp`

### 3.2 每个插件的标准目录

```text
plugins/plugin_name/
├─ CMakeLists.txt
├─ include/
├─ src/
├─ tests/
└─ resources/
```

### 3.3 过滤器命名规则

建议 Visual Studio 中使用以下过滤器：

- Header Files
- Source Files
- Tests
- Resources
- Generated Files

不要把逻辑无关的文件混在一个过滤器中。过滤器只服务于阅读和维护，不应替代真实目录设计。

---

## 4. 构建策略

## 4.1 推荐使用 CMake 统一构建

不建议长期直接手写 `.vcxproj` 作为主维护入口。

原因如下：

- 版本升级时 `.vcxproj` 容易分叉
- 多插件维护时重复配置太多
- 公共属性不容易复用
- CMake 更适合统一管理输出目录、编译选项、依赖和 Maya 版本切换

因此推荐：

- **CMake 作为唯一真实构建源**
- Visual Studio 工程由 CMake 生成

---

## 4.2 mll 插件构建策略

每个插件都应构建为 `MODULE`，并输出为 `.mll`。

应统一做到：

- 后缀为 `.mll`
- 输出目录包含 Maya 版本
- Debug / Release 分开输出
- 构建后可直接复制到测试目录或部署目录

建议输出结构：

```text
out/
└─ maya2026/
   ├─ Debug/
   │  └─ hello_world_mll.mll
   └─ Release/
      └─ hello_world_mll.mll
```

---

## 4.3 构建文件管理规则

- 根目录 `CMakeLists.txt` 只负责全局选项和子目录注册
- 插件自己的 `CMakeLists.txt` 只负责当前插件
- 公共规则全部收敛到 `cmake/` 下
- 不要把 Maya include/lib 路径硬编码在多个插件中

这能明显降低未来 Maya 升级时的修改成本。

---

## 5. 依赖导入规则

## 5.1 Maya SDK 依赖

优先通过环境变量 `MAYA_LOCATION` 管理 Maya 安装路径。

例如：

```bat
set MAYA_LOCATION=C:\Program Files\Autodesk\Maya2026
```

构建系统统一从此变量读取：

- include: `${MAYA_LOCATION}/include`
- lib: `${MAYA_LOCATION}/lib`

## 5.2 第三方依赖规则

本模板默认 **不引入新第三方依赖**。

若未来必须引入：

- 优先使用源码可审查、版本固定的依赖
- 在 `third_party/` 下集中管理
- 记录许可证与版本
- 由管理者批准后再纳入仓库

---

## 6. 项目属性管理

建议管理以下全局属性：

- Maya 版本
- 编译器版本
- C++ 标准版本
- 输出目录
- 警告等级
- 编码与字符集策略
- 是否开启异常/RTTI

推荐默认值：

- C++17
- x64
- Debug / Release 两套配置
- 输出目录按 Maya 版本隔离

不要让各插件自行定义这些全局属性，否则仓库会逐渐失控。

---

## 7. 多项目与多插件维护规则

当同一仓库内存在多个插件时，建议按以下思路维护：

### 7.1 一个插件只负责一类职责

例如：

- `scene_check_mll`
- `mesh_tools_mll`
- `animation_tools_mll`

避免把完全不同职责塞进一个巨大插件中。

### 7.2 公共代码抽到 shared

多个插件共用的内容，例如：

- 日志
- Maya API 包装
- 错误处理
- 字符串转换
- 公共节点/命令注册工具

统一抽到 `shared/`。

### 7.3 版本与发布记录独立维护

建议为每个插件维护：

- 当前版本
- 兼容 Maya 版本
- 输出文件名
- 发布说明

---

## 8. Maya 版本升级流程

从 Maya 2026.3 升级到其他版本时，不建议直接覆盖老配置。

推荐流程：

1. 新增 Maya 版本配置
2. 保留旧版本构建逻辑
3. 在测试机验证编译通过
4. 在 Maya 中完成加载测试
5. 记录 API 变化点
6. 更新兼容矩阵文档
7. 确认无问题后再决定是否清理旧版本支持

### 8.1 升级时要重点检查的内容

- `MAYA_API_VERSION`
- include/lib 路径
- 头文件接口变化
- 链接库变化
- 节点、命令注册相关 API 是否改动
- Python/MEL 侧加载流程是否受影响

---

## 9. 维护建议

### 9.1 每个新插件都从模板复制创建

不要从旧插件随意拷贝并逐个改名。应从统一模板起步。

### 9.2 管理者需要控制以下变更

以下变更应由管理者审批：

- 修改全局 CMake 规则
- 修改输出目录结构
- 新增第三方依赖
- 修改公共 shared 代码结构
- 新增 Maya 版本支持

### 9.3 文档要和模板一起维护

如果项目结构或构建方式发生变化，文档必须同一提交更新。

---

## 10. 最小落地建议

如果当前要先从 0 开始搭建，建议先只做以下内容：

1. 建立一个统一仓库
2. 使用本模板搭建 `cmake/` 与 `plugins/`
3. 先做一个 `hello_world_mll`
4. 跑通编译、加载、执行命令
5. 确认成功后，再开始增加节点类、命令类和部署脚本

这样可以把风险控制在最小范围内，同时为长期维护打下稳定基础。
